# 원리

이 문서는 문서와 워크플로우가 왜 이런 구조를 가지는지 설명해요.

## 1. 티켓 우선 실행

작업은 임의 편집이 아니라 티켓으로 시작해요.

이유:

- 범위가 명시돼요
- 에이전트의 계약이 좁아져요
- 모듈 간 오작업을 줄일 수 있어요

## 2. 서브모듈 격리

모든 작업은 실행 전에 대상 서브모듈을 지정해야 해요.

이유:

- 저장소 안에 서로 다른 규칙을 가진 모듈이 있을 수 있어요
- 서브모듈 단위 범위가 에이전트 동작을 예측 가능하게 해요
- 같은 워크플로우를 다른 프로젝트에도 재사용할 수 있어요

## 3. 변경 전에 설계

설계는 플랜 문서에, 파일 변경은 실행 단계에 둬요.

이유:

- 상태 변경 전에 의도를 검토할 수 있어요
- 승인 대상이 구체적인 설계가 돼요
- 구현 결과를 원래 계획과 비교할 수 있어요

## 4. 정본 규칙 소스

런타임 행동의 정본은 `AGENTS.md`예요.

이유:

- 실행 중인 에이전트는 하나의 권위 있는 파일만 보면 돼요
- 미러 진입점은 얇게 유지할 수 있어요
- IDE가 달라도 워크플로우가 일관돼요

## 5. README는 진입점

README는 모든 규칙을 복제하는 문서가 아니라, 읽는 사람이 방향을 잡는 문서예요.

이유:

- 짧은 진입점이 유지보수하기 쉬워요
- 상세 설명은 별도 문서로 독립적으로 발전할 수 있어요
- 필요한 깊이만 골라 읽게 할 수 있어요

## 6. 국문/영문 정합성

영문과 국문 문서는 같은 워크플로우 모델을 설명해야 해요.

이유:

- 이 저장소는 두 언어로 사용돼요
- 설명이 어긋나면 워크플로우도 흔들려요
- 번역 차이로 인한 해석 오류를 줄일 수 있어요

## 7. 문서는 RAG를 보완해요

문서와 RAG는 같은 문제의 다른 부분을 해결해요.

이유:

- RAG는 과거 결정을 빠르게 찾아줘요
- 문서는 안정적인 모델과 운영 전제를 설명해요
- 둘을 함께 쓰면 해석 오류가 줄어들어요

## 실무 규칙

워크플로우가 바뀌면 README 요약과 상세 원리/구조 문서를 함께 갱신해야 해요.
