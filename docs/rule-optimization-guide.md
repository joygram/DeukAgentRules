# Rule Optimization Guide

How to write `AGENTS.md` and `PROJECT_RULE.md` that AI agents actually follow.

## Why Optimization Matters

LLMs process rules as part of their context window. Poorly structured rules cause:
- **Missed rules**: Buried in prose, the agent skips them.
- **Conflicting interpretation**: Ambiguous phrasing leads to wrong decisions.
- **Token waste**: Verbose descriptions consume context that could hold code.

## Core Principles

### 1. IF-THEN over Prose

```
BAD:  "When modifying files, agents should be careful about whether
       the file might be generated by a build pipeline."

GOOD: CHECK: Is the target file generated?
        → IF @generated or DO NOT EDIT marker → DO NOT EDIT.
        → IF unsure → ASK USER.
```

LLMs parse conditional logic far more reliably than descriptive paragraphs.

### 2. Decision-Point Grouping

Organize rules by **when** the agent encounters them, not by topic:

| Group | When | Example rules |
|-------|------|---------------|
| Boot | Session start | Read AGENTS.md, read PROJECT_RULE.md |
| Pre-Action | Before every tool call | Check active ticket, check if file is generated |
| In-Flight | During execution | Scope creep detection, error handling |
| Post-Action | After completion | Run tests, archive ticket |

### 3. Explicit STOP Signals

Agents tend to continue working unless explicitly told to halt. Use a dedicated HALT table:

```markdown
| # | Condition | Action |
|---|-----------|--------|
| H1 | 2+ repeated errors | Stop → rollback → file ticket. |
| H2 | Scope exceeds boundary | Stop → new ticket. |
```

Numbered halt conditions (H1, H2...) are scannable and referenceable.

### 4. Tables over Bullets for Multi-Field Data

```
BAD:
- Plans go in `.deuk-agent/docs/plans/`
- Reports go in `.deuk-agent/docs/walkthroughs/`

GOOD:
| Type | Path |
|------|------|
| Plan docs | `.deuk-agent/docs/plan/<ticket-id>-plan.md` |
| Archived docs | `.deuk-agent/docs/archive/<YYYY-MM>/<...>.md` |
```

### 5. No Ambiguity Words

Avoid: "such as", "etc.", "things like", "generally", "might", "should consider".
Use: exact file paths, exact conditions, exact actions.

```
BAD:  "Generated directories like dist/, gen/, etc."
GOOD: "Files in dist/ Generated/ gen/ deukpack_out/ = always generated."
```

### 6. Required Fields = CLI Enforcement

Rules that say "MUST include X" are meaningless without enforcement.
Back every required field with a CLI-level validation that **blocks execution** on violation:

```javascript
// In CLI script
const summary = (opts.summary || "").trim();
if (!summary) {
  throw new Error("[VALIDATION FAILED] 'summary' is mandatory.");
}
```

## AGENTS.md Structure Template (v13)

```markdown
## 1. Boot Sequence
  → What to read, what to initialize, 1-CALL ticket discovery

## 2. Pre-Action Guards
  → Table-format checklist (G1-G4) before every tool call

## 3. Ticket Lifecycle
  → Phase table with inline search rules and STOP conditions

## 4. HALT Conditions + File Guards
  → Merged H1-H5 halt triggers + F1-F6 file modification guards

## 5. Docs, Artifacts & Platform
  → Path table, RAG indexing, platform co-existence (merged)

## 6. CLI Reference
  → Command table with max call limits
```

## PROJECT_RULE.md Structure Template

```markdown
---
architecture_docs: "docs/ARCHITECTURE.md"
---

# Project Rules

## Architecture Boundaries
  → Module ownership, dependency direction

## Generated File Mapping
  | Source | Generated | Build command |
  |--------|-----------|---------------|
  → Explicit table so agents know what NOT to edit

## DC-* Guards (project-specific)
  → DC-HALT triggers, DC-INFRA protected zones

## Build & Test Commands
  | Action | Command |
  |--------|---------|
  → Exact commands, no guessing
```

## Anti-Patterns

| Anti-Pattern | Why It Fails | Fix |
|-------------|--------------|-----|
| Long paragraphs | Agent skims, misses critical rules | Use IF-THEN + tables |
| "Be careful about X" | No actionable instruction | "IF X → STOP" |
| Rules scattered across files | Agent may not read all of them | Hub-Spoke: one AGENTS.md hub |
| Optional fields | Always omitted under pressure | CLI enforcement with error |
| Korean+English mix in rules | Inconsistent parsing by multilingual models | Pick one language for rules body |
| Duplicate rules in multiple sections | Conflicting interpretations | Single source of truth per rule |

## Checklist: Is Your Rule File Optimized?

- [ ] Every rule is an action, not a description
- [ ] HALT conditions are in a numbered table
- [ ] Pre-action checks use IF-THEN format
- [ ] No ambiguity words (etc., should, might)
- [ ] Required fields are enforced at CLI level
- [ ] File paths are exact, not examples
- [ ] Rules are grouped by decision point, not topic
- [ ] Total size < 6KB (target for token efficiency)
